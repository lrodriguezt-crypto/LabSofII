🏥 Sistema Piedra Azul

Aplicación de escritorio para la gestión integral de citas médicas, pacientes e historiales clínicos. Opera de forma local sin requerir conexión a internet.

Java 17 · JavaFX · SQLite

👨‍⚕️ Médico / Terapeuta / Agendador

Usuario principal. Registra, modifica y cancela citas. Accede al historial clínico del paciente. Sus acciones están limitadas por su rol en el sistema.

Usuario primario

🧑‍💼 Administrador

Gestiona usuarios del sistema, asigna roles y permisos. Puede crear o desactivar cuentas. Tiene acceso a funciones de configuración del sistema.

Rol administrativo

📄 Archivo CSV

Sistema externo de reportes. El sistema genera archivos CSV con la información de citas para reportes externos y análisis fuera del sistema.

Exportación de datos

✉️ Notificaciones

Sistema externo (correo electrónico o SMS) utilizado para el envío de código de recuperación de contraseña. Integración planificada para iteraciones futuras.

Futuro / Extensión

🖥️ Capa de Presentación

Implementada con JavaFX y archivos FXML. Contiene las vistas de Login, Agenda, Gestión de Pacientes, Historial Clínico y Reportes. Cada vista tiene su Controller asociado siguiendo el patrón MVC.

JavaFX · FXML · Controllers

⚙️ Capa de Negocio

Contiene la lógica de dominio del sistema: validación de credenciales, control de acceso por roles, detección de conflictos de horario, bloqueo por intentos fallidos y generación de reportes CSV.

Java 17 · Servicios · Dominio

🗃️ Capa de Acceso a Datos

Implementa el patrón DAO (Data Access Object) para abstraer las operaciones CRUD sobre la base de datos. Se conecta a SQLite mediante JDBC. Incluye DAOs para Usuario, Cita, Paciente e Historial.

Java · JDBC · Patrón DAO

🗄️ Base de Datos Local

Motor SQLite embebido que almacena toda la información del sistema localmente. No requiere servidor dedicado. Ideal para el entorno con recursos limitados de la clínica.

SQLite · piedra_azul.db
Nivel 3 · Diagrama de Componentes
Zoom hacia el interior de la Capa de Negocio: muestra los componentes individuales (módulos, servicios) que la conforman, sus responsabilidades e interacciones.
Componentes — Capa de Negocio y Módulos del Sistema

🔐 AuthService

Valida credenciales de usuario contra la base de datos (contraseña cifrada). Gestiona el bloqueo por 5 intentos fallidos durante 15 minutos. Coordina con SessionManager para mantener la sesión activa.

BCrypt · Gestión de sesiones

🛡️ RoleAccessControl

Determina qué vistas y acciones están habilitadas para el usuario autenticado según su rol (Médico, Terapeuta, Agendador, Administrador). Aplica la táctica de separación de responsabilidades.

Control de acceso · Roles

📅 AgendaService

Gestiona el ciclo completo de una cita: creación, reprogramación, cancelación. Delega la detección de conflictos de horario al HorarioValidator antes de confirmar cualquier operación.

Lógica de negocio · Citas

✅ HorarioValidator

Verifica que no existan conflictos de horario al registrar o reprogramar citas. Garantiza el escenario de calidad de escalabilidad con respuestas en menos de 3 segundos.

Validación · Disponibilidad

👥 PacienteService / HistorialService

Gestiona la información demográfica y clínica de los pacientes. HistorialService permite registrar y consultar el historial de consultas y tratamientos de cada paciente.

Gestión clínica · Registros

⚙️ UsuarioService / RolService

Componentes administrativos para la gestión de cuentas de usuario y asignación de roles. Solo accesibles para el rol Administrador.

Administración · Roles

Diagrama de Clases — Módulos Auth y Agenda
Esquema de Base de Datos SQLite

🔐 Clase AuthService

Método login(): consulta UsuarioDAO, verifica hash de contraseña, incrementa intentos fallidos, bloquea cuenta tras 5 intentos. Método logout(): limpia la sesión activa.

Java 17 · BCrypt

📅 Clase AgendaService

Método registrarCita(): valida rol del usuario, llama a HorarioValidator, persiste via CitaDAO. Método exportarCSV(): consulta citas con filtros y genera el archivo de salida.

Java 17 · JDBC · CSV

🗄️ Tablas SQLite

El esquema refleja las 5 entidades clave: usuario, rol, cita, paciente e historial_clinico. Las FK garantizan la integridad referencial y las relaciones entre entidades.

SQLite · JDBC

🛡️ Táctica: Disponibilidad

Todos los DAOs usan try-catch con manejo de excepciones SQL. SQLite opera de forma embebida (sin red), garantizando disponibilidad offline. Las operaciones se persisten inmediatamente en disco.

Offline-first · Exception handling
